# Roadmap 003 — Unified Versioning and Automated Releases

| Status   | Date       | Project Version |
|----------|------------|-----------------|
| Draft    | 2026-05-08 | 1.0.1           |

## Problem

New projects scaffolded from ROE carry version information scattered across multiple files — `VERSION`, `package.json`, `pyproject.toml`, `CMakeLists.txt`, firmware headers, and others depending on the stack. This creates several pain points:

- Automated release pipelines must be taught which files to update per project type.
- Manual version bumps risk inconsistency (one file updated, another missed).
- No standard distinction between a regular release and a stable/LTS release, so CI scripts cannot gate behavior on release type.
- Changelogs are written by hand with no enforced structure, making automated aggregation or diffing impractical.
- There is no standard way to identify which version of ROE a project was scaffolded with, making it impossible to assess compliance or determine which ROE features are available in a given project.

## Goal

Provide a single source of truth for version information in every ROE-scaffolded project, with tooling to propagate it, generate changelogs, and cut both standard and stable releases — all from a consistent interface (`make release` / `make release-stable`). Additionally, every project should be able to report the ROE version it was scaffolded with, enabling compliance checks and feature availability queries.

## Proposed Approach

### Single Source of Truth

- The `VERSION` file at the repository root remains the canonical version: `MAJOR.MINOR.PATCH[-PRERELEASE]`.
- A `RELEASE_TYPE` file (or a field inside a `release.toml`) records whether the next release is `standard` or `stable`.
- All other version references (manifests, headers, lockfiles) are **derived** — generated by a `make sync-version` target, never edited by hand.

### Release Types

| Type       | Meaning                                        | Tag format      |
|------------|------------------------------------------------|-----------------|
| `standard` | Regular feature or fix release                 | `vMAJOR.MINOR.PATCH` |
| `stable`   | Long-term / production-hardened release        | `vMAJOR.MINOR.PATCH-stable` |

The release type controls tag suffix and can gate downstream CI behavior (e.g., publish to a `stable` package feed vs. `latest`).

### Automated Changelog

- Adopt [Conventional Commits](https://www.conventionalcommits.org/) as the required commit message format across ROE-scaffolded projects.
- A `make changelog` target generates or updates `CHANGELOG.md` from git history since the last tag, grouped by type (`feat`, `fix`, `chore`, `docs`, `perf`, etc.).
- The changelog entry for each release is stamped with the version, date, and release type.

### Makefile Targets

| Target                | Action                                                                 |
|-----------------------|------------------------------------------------------------------------|
| `make sync-version`   | Propagate `VERSION` to all known manifest files for the project type.  |
| `make changelog`      | Generate/update `CHANGELOG.md` from git log since last tag.            |
| `make release`        | Bump version, sync, generate changelog, commit, tag `vX.Y.Z`.         |
| `make release-stable` | Same as above but tags `vX.Y.Z-stable` and sets `RELEASE_TYPE=stable`.|
| `make release-dry`    | Show what `make release` would do without writing anything.            |
| `make roe-version`    | Print the ROE version this project was scaffolded with.                |
| `make roe-info`       | Print ROE version and the feature set available at that version.       |

### ROE Version Tracking

Each project scaffolded from ROE records the ROE version at scaffold time in a `.roe-version` file at the repository root:

```
1.0.1
```

This file is committed to the project repository and is never modified by `make release` — it reflects the ROE scaffold version, not the project's own version. The ROE version can then be queried at any time:

```
make roe-version        # prints: ROE version: 1.0.1
make roe-info           # prints ROE version + feature flags available at that version
```

This enables:

- **Compliance checks** — tooling or CI can assert that the project meets a minimum ROE version.
- **Feature availability** — scripts can branch on which ROE capabilities are present (e.g., `stable` release support was added in ROE 1.1.0).
- **Audit and reporting** — a portfolio of projects can be scanned to report ROE adoption and version distribution.

The ROE version in `.roe-version` is separate from and independent of the project's own `VERSION` file.

### File Propagation Map

Each project type declares which files receive the version. A `release.toml` (or similar) at the repo root maps targets:

```toml
[version]
source = "VERSION"

[[version.targets]]
file = "package.json"
pattern = '"version": "{version}"'

[[version.targets]]
file = "src/version.h"
pattern = '#define FW_VERSION "{version}"'
```

This makes propagation explicit, auditable, and extensible without modifying the tooling itself.

## Out of Scope

- Monorepo per-package versioning (independent versions per subdirectory) — not needed for current ROE use cases.
- Publishing to package registries — that is a project-level CI concern, not a ROE scaffolding concern.

## Success Criteria

- [ ] `make release` updates `VERSION`, all declared manifest targets, `CHANGELOG.md`, commits, and tags — no manual file edits required.
- [ ] `make release-stable` produces a distinct tag and changelog entry marking the stable designation.
- [ ] A new project scaffolded from ROE has a working `make release-dry` within 5 minutes of initialization.
- [ ] Changelog entries follow a consistent machine-parseable format across all ROE-scaffolded projects.
- [ ] Every ROE-scaffolded project contains a `.roe-version` file that accurately reflects the ROE version used at scaffold time.
- [ ] `make roe-version` and `make roe-info` work without network access and complete in under one second.
- [ ] A portfolio scan script can query `.roe-version` across multiple project repositories and report compliance and version distribution.

## Open Questions

- Should `RELEASE_TYPE` be a file, a field in `release.toml`, or a `make` argument (`make release TYPE=stable`)?
- Which changelog format to standardize on: Keep a Changelog, auto-generated from Conventional Commits, or a hybrid?
- How to handle projects that cannot adopt Conventional Commits (e.g., firmware repos with existing commit conventions)?
- Should `make roe-info` derive feature availability from a static lookup table bundled with ROE, or fetch it from a versioned manifest in the ROE repository?
- What is the minimum ROE version a project must declare to be considered compliant, and how/where is that floor defined?
